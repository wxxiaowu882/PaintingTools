# Solid 页面 — 渲染降级与恢复机制

> 本文档记录 `Solid.html` 的光追渲染降级策略、恢复逻辑和防黑屏保底的设计决策，
> 供后续迭代参考。最后更新：2026-04-15。

---

## 1. 核心问题

Solid 页面使用 `three-gpu-pathtracer`（纯 WebGL2 软件光追）渲染场景。
不同设备/浏览器在以下方面差异巨大：

- GPU 算力：决定每帧 `renderSample()` 的耗时
- Shader 编译速度：首帧可能阻塞 3-60 秒
- 系统资源回收策略：手机后台挂起后 WebGL 上下文会被回收

需要一套**分层**策略，既不误杀有能力的设备，又保证任何设备都不会黑屏。

---

## 2. 三层过滤架构

### 第一层：静态能力检测（同步，< 1ms）

在 WebGLRenderer 创建后、PathTracer 实例化前，检查光追的**硬性前置条件**：

| 检测项 | 作用 | 缺失时的含义 |
|--------|------|-------------|
| WebGL2 上下文 | PathTracer 最低要求 | 设备/浏览器确实不支持 |
| `EXT_color_buffer_float` | Float render target | 光追无法输出到帧缓冲 |
| `OES_texture_float_linear` | 浮点纹理线性过滤 | BVH 查询结果无法正确插值 |

如果不满足，**直接跳过性能测试和光追初始化**，走光栅模式。
这是唯一一种可以确定地说"设备不支持"的情况。

兼容性说明：
- `EXT_color_buffer_float`：Chrome 56+、Firefox 51+、Safari 15+、iOS 15+、华为浏览器（Chromium 内核）均支持
- `OES_texture_float_linear`：Chrome 27+、Firefox 24+、Safari 8+、几乎所有现代设备支持
- 对于不支持 WebGL2 的极老设备（如 iOS 14 以下），Three.js 的 WebGLRenderer 本身就会降级或抛异常，已有兜底

### 第二层：性能测试结果缓存（localStorage）

首次进入时执行 PerfTester，结果写入 `localStorage`：

```
key:   'SolidPerfResult_v1'
value: { passed: boolean, ts: number, ua: string }
```

- 测试通过 → 缓存 `passed: true`。后续访问若 UA 不变且不超过 30 天，跳过测试
- 测试失败 → 缓存 `passed: false`。后续访问仍执行测试（给设备翻身机会），但不缓存通过以外的结果（避免一次失败永远失败）

UA 变化（系统升级、换浏览器）会自动使缓存失效，重新测试。

localStorage 兼容性：所有目标浏览器均支持。隐私模式下写入可能失败，
用 try-catch 包裹，失败时静默回退到每次测试。

### 第三层：运行时保底

#### A. 性能测试失败 → 软降级 + 延迟自动重试

PerfTest 失败不再直接调 `triggerUnsupported()`（判死刑），而是：

1. 切到光栅模式（`renderer.render`），画面正常显示，不黑屏
2. Toast 提示"光追性能不足，已切换为基础模式"
3. 场景加载完毕后等待 12 秒（系统稳定后），静默尝试一次光追重建
4. 重试成功 → Toast "光追已恢复"；重试失败 → 保持降级，不再打扰

#### B. 采样超时保底（防黑屏）

在 `animate()` 光追分支中，如果 `pathTracer.samples === 0` 且距离上次
`reset()` 已超过 8 秒（手机）/ 12 秒（桌面），说明 GPU 连第一帧都渲染不出来：

1. 自动切回光栅模式
2. Toast "渲染超时，已切换为基础模式"
3. 标记 `window._solidSampleTimeout = true`，本次会话不再自动尝试光追
4. 用户仍可手动点"重新渲染"按钮

这比 PerfTest 更实战——PerfTest 用小 Box 测试，通过不代表大场景也跑得动。

#### C. Context Lost 与 PC 端显存（ANGLE / D3D11）— 2026-04-15

**现场特征（PC Chrome / Edge）：** 手动开启光追后出现 `GL_OUT_OF_MEMORY`（HLSL 编译失败），随后 `CONTEXT_LOST_WEBGL`，画面白/黑；恢复逻辑若在**上下文尚未恢复**时反复创建 `WebGLPathTracer`，会必然失败并误触发总降级。

**策略（已实现于 `Solid.html`）：**

1. **`webglcontextlost`**：只做 `preventDefault`、置 `_solidContextLost`、关光追、异步 `dispose` PathTracer，**不再**在丢失瞬间调用 `__solidScheduleRecover('event_context_lost')`（丢失期间 GL 不可用，重建无意义）。
2. **`webglcontextrestored`**：清除 `_solidContextLost` 后调用 `__solidScheduleRecover('event_context_restored')`，在上下文**已恢复**后再尝试 PathTracer。
3. **`__solidScheduleRecover` 内 `run()`**：若 `gl.isContextLost()` 仍为 true，**跳过本轮 PathTracer 重建**并打日志，**不**用无效重试堆满次数去触发 `recover_exhausted`；最后一轮若仍丢失则同样不将「仍丢失」误判为恢复耗尽。
4. **桌面 PathTracer 内部分辨率**：`_solidPathTracerRenderScale()` 对非移动端使用 `min(配置 renderScaleDesktop, 0.64)`，减轻首帧大量 HLSL 编译带来的显存峰值，降低 OOM 概率（略降画质换稳定）。

**说明：** 若 GPU 进程或上下文长时间未恢复，光栅同样无法 `render`，需用户**刷新或重启浏览器**；`[RasterFallback] WebGL context lost` 与 `__solidGetDiagnostics().glContextLost` 用于区分「逻辑降级」与「GL 已死」。

`visibilitychange` 切回前台 → 仍会检测是否需要恢复；光栅模式下额外请求 `_solidRequestRasterEnvProbe('visibility_visible')`，减轻长时间挂起后环境贴图/曝光不一致导致的异常画面（见 §10）。

#### D. 手动停止优先级（2026-04-11 更新）

用户点击“停止渲染”属于明确的人为意图，优先级高于自动恢复：

1. `stopRender()` 会设置 `window._solidUserStoppedRender = true`
2. 在该标记为 true 时，`visibilitychange` 与恢复调度不会自动重启光追
3. 只有用户点击“重新渲染”（`forceReRender()`）时，才会清除该标记并允许恢复

这样可以避免“手动停止后，切后台再回来又自动启动”的体验冲突。

## 5. 状态标志速查

| 标志 | 含义 |
|------|------|
| `window.isUnsupported` | 已降级（静态不支持、或重试全部失败） |
| `window.useAdvancedRender` | 当前是否走光追分支 |
| `window._solidContextLost` | WebGL 上下文已丢失，等待恢复 |
| `window._solidRecovering` | 正在执行恢复尝试 |
| `window._solidSampleTimeout` | 本次会话中光追采样超时过 |
| `window._solidHardwareUnsupported` | 静态检测不通过（真不支持） |
| `window._solidPerfSoftFail` | PerfTest 失败但尚未最终定论 |
| `window._solidUserStoppedRender` | 用户手动停止渲染，自动恢复应让位 |

## 6. 完整决策流程

```
页面加载
  │
  ├─ WebGLRenderer 创建
  │
  ├─ 【第一层·静态检测】
  │   ├── WebGL2? ──否──→ _solidHardwareUnsupported = true → 光栅模式 + "浏览器不支持"弹窗
  │   ├── EXT_color_buffer_float? ──否──→ 同上
  │   └── 通过 → 继续
  │
  ├─ PathTracer 实例化
  │
  ├─ 【第二层·缓存检查】
  │   ├── localStorage 有 passed=true 且 UA 匹配且未过期 → 跳过 PerfTest
  │   └── 否 → 执行 PerfTest
  │       ├── 通过 → 缓存 passed=true，正常渲染
  │       └── 失败 → 软降级（光栅模式 + Toast）
  │           └── 12 秒后静默重试
  │               ├── 成功 → 恢复光追 + 缓存 passed=true
  │               └── 失败 → 保持降级，用户可手动"重新渲染"
  │
  ├─ 【第三层·运行时保底】
  │   ├── animate() 中 samples=0 超 8s/12s → 自动切光栅 + Toast
  │   ├── Context Lost → 释放 PathTracer；恢复仅在 webglcontextrestored / 可见性调度中尝试（丢失期间不盲目重建）→ 仍失败再 triggerUnsupported
  │   ├── 用户手动 stopRender → 标记 userStopped，阻止 visibility 自动恢复
  │   └── animate() catch 连续 3 次非致命异常 → triggerUnsupported
  │
  └── 用户随时可点"重新渲染"手动恢复（会清除 userStopped 标记；硬件真不支持时此按钮隐藏）
```

## 8. 继续定位（控制台取证）

当需要把现场状态交给开发或 Agent 排查时，可在浏览器控制台调用：

| API | 作用 |
|-----|------|
| `__solidGetDiagnostics()` | 返回当前快照对象：`visibility`、渲染/光追相关 `flags`、`pathTracer` 采样与参数、WebGL `VENDOR`/`RENDERER`/扩展、画布尺寸、`shadowProbe` 上次摘要（若跑过 `solidShadowProbeOnce`） |
| `__solidLogDiagnostics()` | 将同一份快照 `JSON.stringify` 后写入页面调试日志（`hwLog`），并返回该对象，便于一键复制 |

快照中额外包含：`lastDegradeReason`（最近一次 `[Degrade]` 的原因与时间戳）、`glContextLost`（当前是否判定为上下文丢失）。若出现「降级后白场 / 光栅不画」，优先看控制台是否出现 `[RasterFallback] WebGL context lost` 或 `[RasterFallback] renderer.render 失败`。

说明：`[Solid]` 普通日志由 `hwLog`/`addHwLog` 同步输出到控制台；`console.error` 仅在与 shader/WebGL/帧缓冲/上下文相关的消息时额外 mirror 到 `hwLog`，避免无关库刷屏。

## 9. PathTracer 与 `EXT_float_blend` 控制台刷屏（iPad 等）

### 现象

Safari / iPad 上 WebGL2 **常不支持** `EXT_float_blend`。`three-gpu-pathtracer` 原先在每帧 `renderSample()` 内调用 `renderer.extensions.get('EXT_float_blend')`。

Three.js 的 `WebGLExtensions.get()` 在扩展为 `null` 时**每次调用仍会 `console.warn`**（`getExtension` 虽会缓存，但 `get()` 里的 warn 在每次 `get` 都会执行），导致**每帧一条**「EXT float blend extension not supported」，长时间光追可堆积成千上万条日志，**严重拖慢 Web Inspector、占用内存与主线程**，极端情况下可加剧卡顿或诱发 watchdog 行为。

### 修复（`docs/js/three-gpu-pathtracer.js`）

将 `supportsFloatBlending()` 改为使用 **`renderer.extensions.has('EXT_float_blend')`**。`has()` 内部走 `getExtension` 缓存路径，**不会**重复触发上述 `console.warn`。

语义等价：仍表示「是否支持浮点混合」；不支持时 PathTracer 继续使用原有 alpha 合成路径。

## 10. iPad 长时间离开后出现「黑场景」— 分析与可做优化

### 可能原因（按常见程度）

1. **WebGL 上下文被系统回收**  
   离开应用或锁屏较久后，iOS 可能挂起或回收 GPU 上下文。回到前台时若未收到可靠的 `webglcontextrestored`，或上下文仍处于 lost，则 `renderer.render` 无效 → **全黑**（页面背景 `#050505` 亦偏黑，观感为「黑场景」）。

2. **光栅环境探针 / PMREM 与真实 GL 状态不一致**  
   长时间后台再回前台，立方体贴图或环境光可能过期，偶发异常渲染或看似全黑；需在「可见」时触发一次环境刷新。

3. **光追 quad 透明度或内部时钟**  
   相对少见；若仅光追模式且 PathTracer 内部状态未与可见性对齐，可能出现短暂黑屏（需结合日志与 `__solidGetDiagnostics` 再判）。

### 已做优化（`Solid.html`）

- **`visibilitychange` → `visible`**：在双 `requestAnimationFrame` 与原有 `invalidateSyncCache`、`lightMoved` / `needsUpdate` 之外，若**非光追**且 GL 未丢失，调用 **`_solidRequestRasterEnvProbe('visibility_visible')`**，使光栅多探针环境在回到前台后有机会重建，减轻「挂起回来画面不对/发黑」的问题。
- 若 **`_solidIsGlContextLost()`** 为 true，打 **`[visibility]`** 日志，提示用户**刷新页面**（与 §8 诊断字段 `glContextLost` 一致）。

### 若仍复现黑屏

- 看控制台是否出现 **`[visibility] 回到前台时 WebGL 上下文仍丢失`** 或 **`[RasterFallback] WebGL context lost`**。若有，属 **GPU/系统层**，应用内无法在不重建 `WebGLRenderer` 的前提下恢复，应提示用户刷新或重启 Safari。
- 后续可增强（未默认开启）：在确认 lost 且 `webglcontextrestored` 长期不来时，展示全屏轻提示「请刷新以恢复画面」；或探索 **重建 WebGLRenderer**（工作量大，需单独排期）。
