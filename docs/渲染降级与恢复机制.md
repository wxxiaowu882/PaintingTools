# Solid 页面 — 渲染降级与恢复机制

> 本文档记录 `Solid.html` 的光追渲染降级策略、恢复逻辑和防黑屏保底的设计决策，
> 供后续迭代参考。最后更新：2026-04-11。

---

## 1. 核心问题

Solid 页面使用 `three-gpu-pathtracer`（纯 WebGL2 软件光追）渲染场景。
不同设备/浏览器在以下方面差异巨大：

- GPU 算力：决定每帧 `renderSample()` 的耗时
- Shader 编译速度：首帧可能阻塞 3-60 秒
- 系统资源回收策略：手机后台挂起后 WebGL 上下文会被回收

需要一套**分层**策略，既不误杀有能力的设备，又保证任何设备都不会黑屏。

---

## 2. 三层过滤架构

### 第一层：静态能力检测（同步，< 1ms）

在 WebGLRenderer 创建后、PathTracer 实例化前，检查光追的**硬性前置条件**：

| 检测项 | 作用 | 缺失时的含义 |
|--------|------|-------------|
| WebGL2 上下文 | PathTracer 最低要求 | 设备/浏览器确实不支持 |
| `EXT_color_buffer_float` | Float render target | 光追无法输出到帧缓冲 |
| `OES_texture_float_linear` | 浮点纹理线性过滤 | BVH 查询结果无法正确插值 |

如果不满足，**直接跳过性能测试和光追初始化**，走光栅模式。
这是唯一一种可以确定地说"设备不支持"的情况。

兼容性说明：
- `EXT_color_buffer_float`：Chrome 56+、Firefox 51+、Safari 15+、iOS 15+、华为浏览器（Chromium 内核）均支持
- `OES_texture_float_linear`：Chrome 27+、Firefox 24+、Safari 8+、几乎所有现代设备支持
- 对于不支持 WebGL2 的极老设备（如 iOS 14 以下），Three.js 的 WebGLRenderer 本身就会降级或抛异常，已有兜底

### 第二层：性能测试结果缓存（localStorage）

首次进入时执行 PerfTester，结果写入 `localStorage`：

```
key:   'SolidPerfResult_v1'
value: { passed: boolean, ts: number, ua: string }
```

- 测试通过 → 缓存 `passed: true`。后续访问若 UA 不变且不超过 30 天，跳过测试
- 测试失败 → 缓存 `passed: false`。后续访问仍执行测试（给设备翻身机会），但不缓存通过以外的结果（避免一次失败永远失败）

UA 变化（系统升级、换浏览器）会自动使缓存失效，重新测试。

localStorage 兼容性：所有目标浏览器均支持。隐私模式下写入可能失败，
用 try-catch 包裹，失败时静默回退到每次测试。

### 第三层：运行时保底

#### A. 性能测试失败 → 软降级 + 延迟自动重试

PerfTest 失败不再直接调 `triggerUnsupported()`（判死刑），而是：

1. 切到光栅模式（`renderer.render`），画面正常显示，不黑屏
2. Toast 提示"光追性能不足，已切换为基础模式"
3. 场景加载完毕后等待 12 秒（系统稳定后），静默尝试一次光追重建
4. 重试成功 → Toast "光追已恢复"；重试失败 → 保持降级，不再打扰

#### B. 采样超时保底（防黑屏）

在 `animate()` 光追分支中，如果 `pathTracer.samples === 0` 且距离上次
`reset()` 已超过 8 秒（手机）/ 12 秒（桌面），说明 GPU 连第一帧都渲染不出来：

1. 自动切回光栅模式
2. Toast "渲染超时，已切换为基础模式"
3. 标记 `window._solidSampleTimeout = true`，本次会话不再自动尝试光追
4. 用户仍可手动点"重新渲染"按钮

这比 PerfTest 更实战——PerfTest 用小 Box 测试，通过不代表大场景也跑得动。

#### C. Context Lost 自动恢复（已实现）

`webglcontextlost` → 释放 pathTracer → 自动 3 次重试恢复 → 失败才降级。
`webglcontextrestored` → 触发恢复调度。
`visibilitychange` 切回前台 → 检测是否需要恢复。

#### D. 手动停止优先级（2026-04-11 更新）

用户点击“停止渲染”属于明确的人为意图，优先级高于自动恢复：

1. `stopRender()` 会设置 `window._solidUserStoppedRender = true`
2. 在该标记为 true 时，`visibilitychange` 与恢复调度不会自动重启光追
3. 只有用户点击“重新渲染”（`forceReRender()`）时，才会清除该标记并允许恢复

这样可以避免“手动停止后，切后台再回来又自动启动”的体验冲突。

## 3. 提示文案分层

| 场景 | 提示 | 位置 |
|------|------|------|
| 静态检测不通过（真不支持） | "当前浏览器不支持光影渲染，可正常观察模型。去电脑端可获得完整体验。" | fallback-msg 弹窗 |
| PerfTest 失败（可能一时拥挤） | "光追暂未启动，已切换基础模式。稍后将自动重试。" | Toast + persistent-warning |
| 采样超时（场景太重） | "渲染超时，已切换为基础模式" | Toast |
| Context Lost 恢复中 | "渲染中断，正在恢复…" | Toast |
| 恢复成功 | "光追已恢复" | Toast 1.5s |
| 最终无法恢复 | "当前设备暂不支持投影和散射光效果，仍可正常观察模型。" | fallback-msg 弹窗 |

## 5. 状态标志速查

| 标志 | 含义 |
|------|------|
| `window.isUnsupported` | 已降级（静态不支持、或重试全部失败） |
| `window.useAdvancedRender` | 当前是否走光追分支 |
| `window._solidContextLost` | WebGL 上下文已丢失，等待恢复 |
| `window._solidRecovering` | 正在执行恢复尝试 |
| `window._solidSampleTimeout` | 本次会话中光追采样超时过 |
| `window._solidHardwareUnsupported` | 静态检测不通过（真不支持） |
| `window._solidPerfSoftFail` | PerfTest 失败但尚未最终定论 |
| `window._solidUserStoppedRender` | 用户手动停止渲染，自动恢复应让位 |

## 6. 完整决策流程

```
页面加载
  │
  ├─ WebGLRenderer 创建
  │
  ├─ 【第一层·静态检测】
  │   ├── WebGL2? ──否──→ _solidHardwareUnsupported = true → 光栅模式 + "浏览器不支持"弹窗
  │   ├── EXT_color_buffer_float? ──否──→ 同上
  │   └── 通过 → 继续
  │
  ├─ PathTracer 实例化
  │
  ├─ 【第二层·缓存检查】
  │   ├── localStorage 有 passed=true 且 UA 匹配且未过期 → 跳过 PerfTest
  │   └── 否 → 执行 PerfTest
  │       ├── 通过 → 缓存 passed=true，正常渲染
  │       └── 失败 → 软降级（光栅模式 + Toast）
  │           └── 12 秒后静默重试
  │               ├── 成功 → 恢复光追 + 缓存 passed=true
  │               └── 失败 → 保持降级，用户可手动"重新渲染"
  │
  ├─ 【第三层·运行时保底】
  │   ├── animate() 中 samples=0 超 8s/12s → 自动切光栅 + Toast
  │   ├── Context Lost → 释放 → 3 次自动重试 → 失败才 triggerUnsupported
  │   ├── 用户手动 stopRender → 标记 userStopped，阻止 visibility 自动恢复
  │   └── animate() catch 连续 3 次非致命异常 → triggerUnsupported
  │
  └── 用户随时可点"重新渲染"手动恢复（会清除 userStopped 标记；硬件真不支持时此按钮隐藏）
```
